# Clarification

## Purpose

This file defines question boundaries and wording rules.

## Boundary

Do not turn clarification into a full requirements interview, PRD, UI design, API design, database design, implementation plan, or test plan.

## Question Rules

When clarification is required:

- ask the fewest questions possible;
- prefer one question per turn;
- if several independent blockers exist, ask one compact checkpoint with at most three decisions;
- label each decision as functional or non-functional when useful;
- provide concrete selectable options and mark one recommended option when safe;
- allow the user to choose, modify, or provide another answer;
- keep context short: no full requirement summary, architecture preview, component list, flow draft, JSON snippet, or diagram preview.

## Sufficiency Gate

Proceed to Step 2 when:

- `project_requirement_inputs.md` says the requirement input is sufficient;
- no unresolved answer would materially change the architecture model or Draw\.io artifact;
- remaining unknowns are implementation details or safe internal defaults.
